
**NOTA SOBRE `ad_performance_id`:** Este campo era obrigatório no banco original mas foi tornado nullable pela migration desta story. Ao inserir campanhas via MOS, **não incluir** `ad_performance_id` no INSERT (ou passar `null`). Dados existentes com FK preenchida não são afetados.

**NOTA SOBRE STATUS:** O campo `status` já existe em `ad_campaigns` mas provavelmente tem valores variados. Para o MOS, padronizar no formulário com: `escalando | testando | pausar | morto`. Dados antigos com outros valores de status serão exibidos como recebidos.

### Cálculo de ROAS no frontend
```typescript
const calculateROAS = (revenue: number | null, spend: number | null): string => {
  if (!revenue || !spend || spend === 0) return '—';
  return (revenue / spend).toFixed(2);
};
```

### Relacionamento com ad_performance
A tabela `ad_campaigns` tem FK `ad_performance_id` → `ad_performance.id`. Para simplificar o MVP:
- Ao criar campanha, o campo `ad_performance_id` pode ser preenchido com um registro existente de `ad_performance` OU pode ser tornando opcional (verificar constraints no banco antes de decidir).
- Alternativa simples: usar `spend` e `revenue` direto em `ad_campaigns` (já existem os campos) sem depender do join com `ad_performance` para o MVP.
